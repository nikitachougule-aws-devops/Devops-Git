## Reset vs Revert vs Rebase

These three Git commands modify commit history in different ways.

| Command        | Purpose                                    | 
| -------------- | ------------------------------------------ | 
| **git reset**  | Undo commits by moving the branch pointer  |
| **git revert** | Undo changes by creating a new commit      | 
| **git rebase** | Move or replay commits onto another branch |

---

## 1. Git Reset

**Reset** moves the current branch to a previous commit. It can also remove commits.

### Types of Reset

### Soft Reset

- Removes the last commit but keeps all changes staged.

```bash
git reset --soft HEAD~1
```

### Mixed Reset (Default)

- Removes the last commit and unstages the changes, but keeps them in your working directory.

```bash
git reset HEAD~1
```

### Hard Reset

- Removes the last commit and permanently deletes all changes.

```bash
git reset --hard HEAD~1
```

**Use Reset When:**

* Undoing local commits.
* Cleaning up commit history before pushing.

---

## 2. Git Revert

**Revert** undoes a commit by creating a new commit that reverses its changes.

```bash
git revert <commit-hash>
```

**Example:**

```text
Before:
A → B → C

Revert C

After:
A → B → C → D
```

* `D` is a new commit that cancels the changes made in `C`.
* The original commit remains in the history.

**Use Revert When:**

* The commit has already been pushed.
* You want to safely undo changes without rewriting history.

---

## 3. Git Rebase

**Rebase** moves commits from one branch and places them on top of another branch, creating a cleaner, linear history.

```bash
git switch feature-login
git rebase main
```

**Example:**

Before Rebase:

```text
A → B → C (main)
     \
      D → E (feature)
```

After Rebase:

```text
A → B → C → D → E
```

* The feature branch is updated as if it was created from the latest `main`.
* Commit history becomes cleaner.

**Use Rebase When:**

* Updating a feature branch with the latest changes from `main`.
* Keeping project history clean and linear.

**Avoid rebasing commits that have already been shared with others**, because it rewrites commit history.

---

## Summary

| Feature              | Reset      | Revert              | Rebase               |
| -------------------- | ---------- | ------------------- | -------------------- |
| Undoes commits       | ✅          | ✅                   | ❌                    |
| Creates a new commit | ❌          | ✅                   | ❌                    |
| Rewrites history     | ✅          | ❌                   | ✅                    |
| Safe after pushing   | ❌          | ✅                   | ❌                    |
| Best for             | Local undo | Shared repositories | Clean commit history |

---

## Key Points

* **`git reset`** removes commits by moving the branch pointer. Use it for local changes before pushing.
* **`git revert`** safely undoes a commit by creating a new commit. It is the preferred option for shared repositories.
* **`git rebase`** moves commits onto another branch to create a clean, linear commit history. Avoid using it on commits that have already been pushed and shared.
